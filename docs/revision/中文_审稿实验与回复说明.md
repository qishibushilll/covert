## 4. 隐蔽性 / 可检测性实验

### 目的

回应审稿人最核心的问题：

> 这些弹幕到底像不像正常弹幕？会不会被统计检测器抓出来？

### 脚本

```text
detectability_baseline_test.py
```

### 运行命令

```powershell
& 'C:\Users\15052\AppData\Local\Programs\Python\Python313\python.exe' -X utf8 .\detectability_baseline_test.py
```

### 实验怎么做

实验构造两类弹幕：

1. benign comments  
   从普通弹幕模板和干扰弹幕模板中随机生成。

2. covert comments  
   用当前 sender 真实生成携带协议片段的弹幕流。

然后从每条弹幕提取特征：

- 弹幕长度 `length`
- 特殊符号载体数量 `carrier_symbol_count`
- 标点载体数量 `carrier_punctuation_count`
- 空格数量 `whitespace_count`
- 最大连续空格长度 `max_space_run`
- 是否以 4 个特殊符号结尾 `ends_with_four_symbols`

检测器有两个：

1. 规则检测器

```text
如果弹幕末尾连续 4 个特殊符号，则判定可疑。
```

2. z-score 统计检测器

先统计普通弹幕的特征均值和标准差，再看 covert 弹幕偏离普通弹幕有多远。

### 两种检测视角

#### loaded-only

只看真正携带协议片段的弹幕。

这是强攻击者视角，假设对方已经能筛出可疑弹幕。

#### stream-level

看完整弹幕流，包括载荷弹幕和普通干扰弹幕。

这是普通观察者或平台整体统计视角。

### 当前结果

```text
Profile: loaded-only
loaded_ratio=0.3333
length: 7.9920 vs 12.3300
carrier_symbol_count: 0.0000 vs 1.2160
carrier_punctuation_count: 0.6380 vs 4.0200

Rule detector F1 = 0.0000
Z-score detector F1 = 1.0000
length_js = 0.7957
carrier_symbol_count_js = 0.5605
```

解释：

- 固定四符号规则已经抓不到了，所以 `Rule detector F1 = 0.0000`。
- 但是 loaded-only 仍然非常容易被统计检测器抓出来，`Z-score F1 = 1.0000`。
- 原因是载荷弹幕更长，载体符号和标点更多。

```text
Profile: stream-level
loaded_ratio=0.3333
length: 7.9920 vs 9.3040
carrier_symbol_count: 0.0000 vs 0.4120
carrier_punctuation_count: 0.6380 vs 1.8640

Rule detector F1 = 0.0000
Z-score detector F1 = 0.6667
length_js = 0.1226
carrier_symbol_count_js = 0.1391
```

解释：

- 混入普通弹幕后，整体弹幕流的异常程度下降。
- 规则检测器仍然完全失效。
- 统计检测器仍有一定效果，但比 loaded-only 弱很多。

### 怎么回复审稿人

不能写：

```text
Our method is undetectable.
```

应该写：

```text
The proposed method removes the fixed four-symbol suffix pattern and reduces stream-level statistical deviation. However, loaded-only comments remain distinguishable under a strong statistical detector. Therefore, the proposed method should be understood as a configurable robustness-overhead-concealment trade-off rather than a perfectly undetectable channel.
```

中文解释：

> 我们去掉了最明显的固定四符号模式，并降低了整体弹幕流层面的统计偏差。但是，如果攻击者能单独筛出载荷弹幕，仍然可以检测出来。因此本文不声称完全不可检测，而是报告鲁棒性、开销和隐蔽性之间的权衡。

## 5. 原始固定符号方案为什么不行

### 原始问题

原始 sender 把 4 位协议码直接映射成 4 个特殊符号，放到弹幕末尾。

示例：

```text
主播好厉害●◆■▲
这个操作太秀了▼▶●◆
```

### 检测结果

固定四符号规则检测器可以直接抓：

```text
Rule detector F1 = 1.0000
```

也就是说，原始版本几乎一眼假。

### 我们怎么改

1. 不再使用固定 4 个特殊符号后缀。
2. 把 4 位协议码拆成两个 2 位短片段。
3. 使用标点和符号逐位混合载体。
4. 每个载荷弹幕后插入普通弹幕，降低载荷密度。
5. 后续为了抗丢包，又加入序号和重复发送。

## 6. 一位片段实验

### 目的

测试是否可以把 2 位片段进一步拆成 1 位片段，以降低单条载荷弹幕异常。

### 结果

#### 一位片段，无边界符

优点：

- loaded-only 统计检测 F1 降到约 `0.8256`。

缺点：

- 接收端会把普通弹幕末尾的 `？/！/～` 误判成协议片段。
- 干净闭环失败。

#### 一位片段，有边界符

优点：

- 闭环恢复。

缺点：

- 边界符本身变成新检测特征。
- loaded-only F1 又升到约 `0.9618`。
- 片段数量翻倍，开销变大。

### 结论

一位片段暂时不适合作为主线协议。

主线保留：

```text
PROTOCOL_FRAGMENT_SIZE = 2
```

## 7. 序号化冗余片段实验

### 目的

解决 10% 弹幕丢失下解码失败的问题。

原来的短片段方案依赖片段顺序。如果丢了一条弹幕，后面的片段会错位。

### 改法

每个片段携带：

```text
seq(2 digits) + frag_idx(1 digit) + fragment(2 digits)
```

例如：

```text
03012
```

含义：

- `03`：第 3 个协议码
- `0`：第 0 个片段
- `12`：片段数据

每个片段重复发送：

```python
FRAGMENT_REPLICAS = 3
```

receiver 端：

1. 提取所有片段记录。
2. 按 `seq` 分组。
3. 按 `frag_idx` 分组。
4. 多数投票。
5. 重组 4 位协议码。

### 结果

加入序号化冗余之后：

```text
default 10% loss success = 0.9200
```

在 trade-off sweep 中：

```text
FRAGMENT_REPLICAS=3, FILLERS_PER_PAYLOAD=2
success_10pct = 0.9667
avg_comments = 361.00
loaded_ratio = 0.3333
```

### 怎么解释

序号化冗余显著提高了抗丢包能力。

但代价是：

- 弹幕数量增加。
- 每条 loaded comment 携带更多元数据。
- loaded-only 检测更容易。

所以这是一个鲁棒性和隐蔽性的权衡。

## 10. PQ / 后量子部分怎么处理

你之前说 PQ 代码风险暂时不用管，所以这里不展开改代码。

但回复审稿人时需要注意措辞。

不要过度强调“每条消息都完整携带 ML-KEM 密文”。

建议说：

```text
Ordinary messages use lightweight session advancement, while post-quantum resynchronization is periodic. The resynchronization payload is fragmented using the same protocol packetization mechanism as ordinary payloads, and its overhead is amortized over the rekey interval.
```

中文意思：

> 普通消息不每次都带完整 PQ 材料，而是使用轻量状态推进。PQ 重同步是周期性的，重同步负载也使用同样的协议分片机制，并且开销按重同步间隔摊销。

## 14. 直播间自适应弹幕外壳实验

### 目的

进一步解决固定模板的问题。

原来 sender 使用固定模板库，例如：

```text
主播好厉害
这个游戏好玩吗
大家晚上好
```

长期运行时，这种固定模板会产生两个风险：

1. 句子重复率高。
2. 长度和标点分布不像当前直播间。

所以新增“直播间自适应弹幕外壳”实验。这里的重点不是只给某一个直播间准备一个固定 `room_comments.txt`，而是形成一个通用流程：

> 无论进入哪个直播间，发送端都先观察该直播间最近的真实弹幕，学习这个房间的长度、标点、常见短语和语气风格，生成当前房间专属模板，然后再用这些模板作为隐蔽通信外壳。

也就是说，模板不是离线写死的，而是在通信前根据目标房间动态生成。

### 新增脚本

```text
room_adaptive_stealth_experiment.py
room_style_learner.py
```

其中：

- `room_style_learner.py`：真实直播间风格学习脚本。
- `room_adaptive_stealth_experiment.py`：验证学习式外壳是否降低统计偏差的实验脚本。

### 运行命令

```powershell
& 'C:\Users\15052\AppData\Local\Programs\Python\Python313\python.exe' -X utf8 .\room_adaptive_stealth_experiment.py
```

### 任意直播间怎么先学习

对任意直播间，例如房间号 `ROOM_ID`，先运行：

```powershell
& 'C:\Users\15052\AppData\Local\Programs\Python\Python313\python.exe' -X utf8 .\room_style_learner.py --room ROOM_ID --target-count 80 --rounds 12 --sleep 5
```

脚本会生成：

```text
room_profiles/room_ROOM_ID_comments.txt
room_profiles/room_ROOM_ID_templates.txt
room_profiles/room_ROOM_ID_profile.json
```

并默认把本次学习到的模板激活为：

```text

```

因此 sender 后续会自动读取该房间的学习模板。

如果不想覆盖当前激活模板，可以加：


## 16. 紧凑内嵌载体映射：继续提升 loaded-only 隐蔽性

### 目的

上一轮结果说明，stream-level 隐蔽性已经明显改善，但 loaded-only 视角仍然是最大短板。原因是每条载荷弹幕仍带有较明显的结构化片段：

```text
seq(2 digits) + frag_idx(1 digit) + fragment(2 digits)
```

旧版本把这 5 位记录直接映射成 5 个混合载体字符，并基本作为尾部片段出现。即使固定四符号后缀规则已经失效，统计检测器仍然能从长度、载体数量和特殊符号计数中识别 loaded comments。

### 改法

新增紧凑内嵌载体映射：

```text
5 位十进制片段记录 -> base-12 紧凑值 -> 4 个常见标点载体
```

发送端默认启用：

```python
COMPACT_EMBEDDING_ENABLED = True
COMPACT_CARRIER_ALPHABET = "，。！？；：、～…—,."
COMPACT_RECORD_SIZE = 4
```

特点：

1. 协议逻辑不变，仍然是 `seq + frag_idx + fragment`。
2. 冗余、投票和序号化重组逻辑不变。
3. 载体字符从 5 个减少到 4 个。
4. 不再使用 `●◆■▲` 这类特殊符号作为默认 loaded carrier。
5. receiver 同时兼容新 compact 格式和旧 trailing mixed 格式。

如果需要回退旧格式，可以设置：

```powershell
$env:COVLBCG_COMPACT_EMBEDDING='0'
```

### 怎么写进论文 / 回复

可以写：

```text
We further replace the legacy five-character mixed-carrier suffix with a compact in-text carrier mapping. The logical sequence-indexed fragment record is unchanged, but it is packed into four common punctuation carriers and embedded into the wrapper text. This reduces loaded-comment length and removes special-symbol carrier usage while preserving the same loss-recovery behavior. The improvement is incremental: loaded-only comments remain distinguishable, so the result is reported as a concealment improvement rather than a proof of undetectability.
```

中文解释：

> 我们进一步把旧的 5 字符混合载体后缀改为紧凑内嵌标点映射。协议记录本身不变，但被压缩为 4 个常见标点并嵌入外壳文本中。这样降低了 loaded comment 长度，去掉了特殊符号载体，并保持原有抗丢包恢复能力。但 loaded-only 仍然可检测，所以它只是隐蔽性增量改进，不是彻底解决。

### 下一步提升空间

当前 compact in-text 只是“位置和长度”上的改进，还没有真正理解语义边界。下一步最有价值的是：

```text
semantic punctuation placement / template-level punctuation rewriting
```

也就是让 4 个载体标点落在自然的断句位置，例如语气词后、短句边界、表情 token 附近，而不是简单按字符位置插入。这样才可能进一步压低 loaded-only 检测。
